# Project Overview

## Product Vision

**Secure Freelance Marketplace** — це платформа для безпечної організації freelance-роботи між користувачами. Один користувач може одночасно виступати як замовник і як виконавець: створювати власні задачі, переглядати задачі інших користувачів, подавати пропозиції та виконувати замовлення.

Основний акцент проєкту — безпечна передача даних, захист файлів, контроль цілісності, цифровий підпис критичних подій і escrow-механізм, який зменшує ризик шахрайства між сторонами.

## User Roles

У системі є один базовий тип звичайного акаунта — `user`. Він може виконувати різні дії залежно від контексту:

- `client` — користувач, який створив task і оплачує роботу.
- `freelancer` — користувач, який прийняв task і виконує роботу.
- `moderator` — привілейований користувач, який розглядає disputes.
- `admin` — привілейований користувач для системного адміністрування.

Ролі `client` і `freelancer` не є окремими типами акаунтів. Вони визначаються участю користувача в конкретному `task` або `contract`.

## Core Business Flow

1. Користувач створює task із назвою, описом, ціною та вимогами.
2. Інші користувачі переглядають відкриті tasks.
3. Зацікавлений користувач відкриває task і створює offer.
4. Автор task і потенційний freelancer можуть обговорити деталі через encrypted communication.
5. Якщо сторони домовились, обидві сторони підтверджують співпрацю.
6. Task переходить у стан contract.
7. Client вносить кошти в simulated escrow.
8. Freelancer виконує роботу і передає encrypted files.
9. Client перевіряє результат і підтверджує завершення.
10. Escrow release переводить кошти freelancer-у.
11. Якщо client не підтверджує завершення або виникає конфлікт, створюється dispute.
12. Moderator переглядає матеріали й ухвалює рішення: release або refund.

## Key Product Rules

- Один користувач не може прийняти власну task як freelancer.
- Ціну task можна змінювати тільки до створення contract.
- Contract створюється тільки після підтвердження обох сторін.
- Escrow hold має бути створений до старту роботи.
- Release коштів можливий після підтвердження client або рішення moderator.
- Повідомлення та файли передаються у зашифрованому вигляді.
- Критичні події фіксуються в audit log.

## MVP Scope

MVP фокусується на backend API, структурі даних і security architecture. Frontend може бути реалізований пізніше як окремий web client або admin interface.

У MVP не планується інтеграція з реальними платіжними системами. Замість цього використовується simulated escrow, який дозволяє показати бізнес-логіку без юридичних і фінансових ризиків.

## Future Improvements

- WebSocket communication для realtime-чату.
- Інтеграція з payment provider через webhook-based flow.
- Репутаційна система та рейтинги.
- OAuth login через Google або GitHub.
- Admin dashboard для moderation та audit review.
- S3/MinIO production storage для файлів.
